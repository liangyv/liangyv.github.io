    第24章 最佳实践

    # 可维护性

    # 什么是可维护的代码

    • 可理解性
    • 直观性
    • 可适应性
    • 可扩展性
    • 可调试性

    # 代码约定

    需要加注释的地方：

    • 函数和方法
    • 大段代码
    • 复杂的算法
    • Hack

    命名应该：

    • 变量名应该为名词
    • 函数名应该以动词开始
    • 命名要合乎逻辑和可以理解，不要担心太长

    变量类型透明，js类型是松散的，所以可有有以下三种方式来标识：

    1. 初始化时给一个默认值
    2. 匈牙利标记法，给变量加一个定义的开头
    3. 类型注释

    # 松散耦合

    1. 解耦HTML和JS
    2. 解耦CSS和JS
    3. 解耦应用逻辑和事件处理逻辑

    应用和业务逻辑之间松散耦合的几条原则：

    • 勿将event对象传给其他方法，只传来自event对象中所需要的数据
    • 任何可以再应用层面的动作都应该可以在不执行任何事件处理程序的情况下进行
    • 任何事件处理程序都应该处理事件，然后将处理结果转交给应用逻辑

    # 编程实践

    1. 尊重对象所有权
    2. 避免全局变量
    3. 避免与null比较
    4. 使用常量

    # 性能

    # 注意作用域

    1. 避免全局查找
    2. 避免with语句

    这2个问题都可以通过局部变量的方式将全局查找的次数缩小，从而提升执行速度。

    # 选择正确方式

    避免不必要的属性查找

    标记 名称 描述
    O(1) 常数 不管值有多少，执行时间都是恒定的
    O(log n) 对数 执行时间和值的数量有关，但是不一定需要执行每个值。比如：二分查找
    O(n) 线性 执行时间和值的数量直接相关。比如：遍历
    O(n²) 平方 执行时间和值的数量有关，并且每个值都要执行多次。比如：插入排序
    • 常量、变量、数组直接下标取值的的复杂度都是O(1)
    • 属性查找的复杂度是O(n)

    优化循环

    1. 减值迭代（其实目的就是不要重复计算数据长度，增值的话在循环外部获得确定的循环次数常量也是一样的）
    2. 简化终止条件
    3. 简化循环体
    4. 使用后测试循环（避免最初终止条件计算）

    展开循环

    当循环次数确定时，可以完全展开循环内容，直接书写代码。

    当循环次数不定时，可以通过Duff装置，讲循环用switch-case按照8的倍数去执行。但是这个方法主要是处理大数量，如果数据量较小得不偿失。

    var iterations = Math.floor(values.length / 8);
var leftover = values.length % 8;
var i = 0;

if(leftover > 0){
	do{
		process(values[i++]);
	}while(--leftover > 0);
}

do{
	process(values[i++]);
	process(values[i++]);
	process(values[i++]);
	process(values[i++]);
	process(values[i++]);
	process(values[i++]);
	process(values[i++]);
	process(values[i++]);
}while(--iterations > 0);

    避免双重解释

    避免用eval()函数、new Fuction()、setTimeout等去执行js字符串，而是应该直接声明，避免js解析器启动解析的开销

    其他注意事项

    1. 原生方法较快：原生方法特别是数学计算都是用C/C++实现的
    2. Switch语句较快：替换大量if-else，还可以通过case顺序把最可能被执行的放到最前来加速
    3. 位运算较快：取模、逻辑与、逻辑或都可以考虑用位运算来代替

    # 最小化语句数

    1. 多个变量声明：能一起声明的就一起声明
    2. 插入迭代值：把i++这类操作合并
    3. 使用数组和字面量

    # 优化DOM交互

    1. 最小化现场更新
    2. 使用innerHTML
    3. 使用事件代理
    4. 注意HTMLCollection

    何时会返回HTMLCollection对象：

    • getElementsByTagName()调用
    • 获取了元素的childNodes属性
    • 获取了元素的attributes属性
    • 访问了特殊集合，比如document.forms、document.images等

    # 部署

    # 构建过程

    为什么需要构建过程：

    • 保证自己的代码知识产权，不能轻易暴露代码意图
    • 降低文件大小，去掉无意义的代码和分隔符以及减小冗余的变量名
    • 重新组织代码

    # 验证

    文章提供的方案比较老了，现在一般都是用ESLint来做了

    # 压缩

    JS文件压缩，其实是在讨论2个问题：

    1. 代码长度：浏览器需要解析的字节数
    2. 配重(Wire weigth)：实际从服务器传递到浏览器的字节数

    压缩方式：

    1. 文件压缩：删除空白符、换行符，删除注释，缩短变量名
    2. HTTP压缩：Web服务器开启压缩，浏览器解压缩，例如开启gzip压缩
